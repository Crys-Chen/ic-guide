# Contributing

If you've landed here, you're willing to spend time helping a stranger save themselves some detours — thank you.

This site is essentially a *knowledge map*: which courses are worth taking, what each research direction is actually about, who the people doing the work are. The more complete and accurate the map, the fewer wrong turns the people coming after take. But one person can't draw the whole map — I've only spent four years in the microelectronics department at Fudan, and I can only see clearly the road I personally walked. **The real quality of this site depends on whether others, who walked different roads, are willing to add their stretches.**

## What you can contribute

### 1. Fix mistakes

Lowest-cost, highest-impact contribution. If you find:

- **Wrong professor info** (name, URL, inaccurate research direction, person has left the institution, etc.)
- **Dead course links** or videos that have been removed
- **Typos / unclear sentences**
- **Factual claims in a research-direction page that contradict what you know to be true** (e.g. a company's current status, a technology timeline)

Just file a [GitHub Issue](https://github.com/Crys-Chen/Fudan-ME/issues). One sentence is fine. Even "Prof. X's link is 404" is welcome.

### 2. Add missing professors

If there are notable professors in your area that this site missed — especially Chinese mainland researchers outside Tsinghua/Peking/Fudan, overseas Chinese representatives, or junior faculty working on something interesting — feel free to add them. **The one hard rule is no fabrication**:

- The homepage URL must load, and the page must actually be the professor in question (not a same-name different person, not a department/group homepage)
- The "sub-direction" description must come from their personal homepage / Google Scholar / lab page — don't guess from name + field
- Overseas Chinese profs use `English Name (Chinese Name)` format; HK profs with Cantonese romanization should use Traditional Chinese; if you can't find a verified Chinese name, leave it out

Copy an existing entry and modify.

### 3. Expand course pages

If you've taken a course deeply, done its labs/projects, feel free to expand or add a course page. Course pages follow the seven-field structure in [`template.md`](https://github.com/Crys-Chen/Fudan-ME/blob/main/template.md) (course number / institution / prerequisites / language / difficulty / hours / resources).

When writing —

- **Don't restate the syllabus**. We want the *been-there* perspective: why this course is worth taking / where it loses people / what specifically pays off / what you can actually do after finishing. Three hundred well-chosen words beat three thousand words of TOC.
- **Less sentiment, more substance**. "I learned a lot" tells the reader nothing; "the cache-miss lab is where I first developed intuition for the memory hierarchy" is useful.
- **Don't false-balance**. If a course is hard, say it's hard; if a professor's lectures are weak, say so (but **separate institutional critique from personal attack** — criticizing a curriculum design is fair, naming and shaming an individual instructor is not).

### 4. Expand research-direction pages

If you've done serious research in one of the listed directions (or in a direction we should list but currently don't), feel free to expand. Research-direction pages have a seven-section skeleton:

1. **What this direction is studying** (tell a story; history, motivation, not a pile of terms)
2. **Who this fits** (specific: background, temperament, whether you'll stick with it)
3. **Core research questions** (3–5 items, one sentence each)
4. **Representative institutions** (companies + top conferences + domestic/international)
5. **Knowledge path** (mermaid graph linking to course pages on this site)
6. **Three-step onboarding** (specific: which book's which chapter, which course's which lab)
7. **Related research groups** (ordered: Tsinghua → Peking → Fudan → other domestic → HK → overseas)

The hardest and most valuable section is the first — explaining a direction *to someone outside the direction*, with concrete cases ("the AlphaGo engineers were facing this problem...") not stacks of jargon.

### 5. Engineering-tools tutorials

If you've deeply used Vivado / Cadence / Spectre / KiCad / Gem5 / VSCode / LaTeX / Vim, etc., consider writing a "quick start + my own gotchas" style tutorial. This kind of content has been the site's biggest gap.

## How to contribute

By the size of your change / your Git proficiency, three paths:

### Path A: Small change (recommended for 90% of contributors)

Just [open an Issue](https://github.com/Crys-Chen/Fudan-ME/issues/new) describing what you'd change. One sentence is enough. I'll handle it.

### Path B: Comfortable with Git, larger change

Standard GitHub fork + PR:

1. Fork [Crys-Chen/Fudan-ME](https://github.com/Crys-Chen/Fudan-ME) to your account
2. Clone locally, create a branch
3. Edit — if you add a new page, also add a line under `nav:` in `mkdocs.yml`, otherwise it won't appear in navigation
4. Install `mkdocs-material` and the deps in `requirements.txt`, run `mkdocs serve` to preview
5. Push to your fork, open a PR

Use a one-line PR title (e.g. `feat: add 5 overseas profs to silicon photonics` or `fix: update MIT 6.S081 link`).

### Path C: Don't know Git / want to discuss first

Email me, or @ me in an Issue. For larger contributions (a whole new direction, multi-thousand-word course pages), discuss the direction first — saves the case of writing something only to find it doesn't fit.

## Style conventions

The site has its own *voice*. If your contribution is content (.md files), please follow these:

- **Chinese-primary, keep English terms** (write ADC, TPU, Transformer as-is; common Chinese terms like "加速器" stay Chinese)
- **No padding phrases**. "In summary", "To conclude", "In today's rapidly evolving X" — delete entire paragraphs of these, nothing of substance is lost
- **Take a stance**. "This course's lab environment is over-engineered and I don't recommend self-study" is far more useful than "this course has a certain level of challenge"
- **Critique systems, not individuals**. Criticizing curriculum design, exam culture, industry status quo is fine; naming-and-shaming a specific instructor is not
- **No AI flavor**. If you drafted with an LLM, **rewrite it yourself** before submitting. The "smooth transitions" and "perfect summaries" LLMs produce are exactly the tone this site rejects
- **No emoji**, unless content-specific (showing emoji as content, etc.)

## A note

This site doesn't accept sponsorship, ads, or paid placement of any kind. Contributions with implicit commercial intent (promoting a vendor's tool, a paid course) will be rejected.

If you want your name on the site (contributor list / commit author), great; if you'd rather stay anonymous, also fine — this site was never about collecting names.

Finally: **the bar to contributing is low, but the bar for contributions is high**. A typo fix is welcome; an entire research-direction guide is welcome. What matters is *accurate, useful, with a perspective* — not *more*.

Thanks for being willing to draw your road onto this map.
