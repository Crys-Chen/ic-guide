    参与建设

    如果你点进这里，说明你愿意花时间帮一个素未谋面的同学少走一段弯路——谢谢。

    这个站本质上是一份"知识地图"：哪些课程值得看、哪些科研方向在做什么、哪个老师在做什么。地图越全、越准，越多后来者能少走弯路。但一个人画不出完整的地图——我只在复旦微电子待过四年，能看清的只是我自己走过的那条路。这个站的真正质量取决于其他走过别的路的人愿不愿意把自己那段补上。

    你能贡献什么

    一、改错

    最低成本，最大收益的贡献。如果你发现：

    • 教授信息错了（名字、URL、研究方向描述不准、人已经离职等）
    • 课程链接死了或视频被删了
    • 错别字、语义不通的句子
    • 科研方向页里某段事实陈述与你了解的实际情况不符（比如某个公司近况、某项技术发展节点等）

    直接提 GitHub Issue 说一句就行。哪怕只是"XX 老师的链接 404 了"也欢迎。

    二、补充教授

    如果你的方向里有这个站漏掉的重要教授（尤其是清北复以外的国内学者、海外华人代表、相关方向的青年教授），欢迎补充。唯一的硬约束是不能编：

    • 教授主页 URL 必须能打开，且页面确实是他/她本人的（不是同名同姓的另一个人，不是学院/课题组主页）
    • "子方向描述"必须来自其本人主页 / Google Scholar / 课题组页可见的内容，不能凭名字+方向猜
    • 海外华人按"英文名（中文名）"格式；粤语拼音的港校教授中文名用繁体；查不到中文名就只保留英文名

    格式见现有 entry，复制一条改即可。

    三、扩充课程页

    如果你深度学过一门课、做过它的 lab/project，欢迎补充课程页或新增课程页。课程页结构参考 template.md 的七字段（课号 / 所属大学 / 先修 / 编程语言 / 难度 / 学时 / 资源汇总）。

    写课程页时——

    • 避免变成讲义复述。我们要的是"过来人的视角"：这门课为什么值得学 / 哪里容易劝退 / 哪里值得花时间 / 学完应该能做什么。一段两段三百字胜过三千字的目录大纲。
    • 少抒情、多干货。"学完受益匪浅"对读者没用；"这门课讲 cache miss 的实验是我对内存层次有直觉的转折点"才有用。
    • 不要"为平衡而平衡"。课难就说难，老师不行就说老师不行（但对系统的批评和对人的批评要分开——批培养方案不等于苛责某个具体的老师）。

    四、扩充科研方向

    如果你在站点列出的某个方向（或者你认为应该列但目前没列的某个方向）做过认真的研究，欢迎补充。科研方向页的"七段骨架"是：

    1. 这个方向在研究什么（讲故事，讲历史，讲动机，不要堆术语）
    2. 适合什么样的人（具体到背景、性格、能不能坚持）
    3. 核心研究问题（3-5 条，每条一句话）
    4. 代表性机构（企业 + 顶会 + 国内/国际）
    5. 知识路径（mermaid 图，链到本站已有课程页）
    6. 入门三步走（具体到读哪本书的哪一章、跟哪门课的哪个 lab）
    7. 相关课题组（按"清华 → 北大 → 复旦 → 其他国内 → 港校 → 海外"排序）

    写新方向最难也最有价值的是第一节——把一个方向用"非这个方向的人也能看懂的话"讲清楚。多用具体案例（"AlphaGo 的工程师当年面对的是这样一个问题..."）、少用堆叠的术语。

    五、工程工具教程

    如果你深度用过 Vivado / Cadence / Spectre / KiCad / Gem5 / VSCode / LaTeX / Vim 等工具，欢迎写一篇"快速上手 + 自己踩过的坑"风格的教程。这类内容站内一直最缺。

    怎么贡献

    按你的改动规模 / 对 Git 的熟悉程度，三条路径：

    路径 A：很小的改动（推荐 90% 的贡献者用）

    直接 开一个 Issue，描述你想改什么。一句话也行。我会处理。

    路径 B：会用 Git，改动较大（新增课程页 / 扩充科研方向）

    标准 GitHub fork + PR 流程：

    1. Fork Crys-Chen/Fudan-ME 到你的账号
    2. 本地 clone，新建一个 branch
    3. 改文件——如果新增页面，记得同步在 mkdocs.yml 的 nav: 里加一行链接，否则页面不会出现在导航里
    4. 本地装 mkdocs-material + requirements.txt 里的依赖，跑 mkdocs serve 看下渲染效果
    5. push 到你的 fork，发 PR

    PR 标题用一句话讲清楚改了什么（比如 feat: 补充硅光集成方向 5 位海外教授 或 fix: 更新 MIT 6.S081 链接）。

    路径 C：不会 Git / 想先聊聊

    直接给我邮件或在 Issue 里 @ 我。如果是大块内容（一整个新方向、几千字课程页），可以先聊聊定方向，免得写完发现不对路。

    写作风格约定

    这个站有自己的"声音"。如果你的贡献是要写成站内 .md 内容，麻烦看一下下面几条——

    • 中文为主，英文术语保留（写 ADC、TPU、Transformer 不必翻译；但常见词如"加速器"就别写 accelerator）
    • 不要套八股。"综上所述"、"总而言之"、"在 X 飞速发展的今天"——这些句子整段删掉对内容没有任何损失
    • 观点可以鲜明。"这门课对实验环境的要求过高、不建议自学"比"这门课有一定挑战性"有用得多
    • 批制度，不苛责个人。批评培养方案、考试制度、行业现状是 OK 的；指名道姓骂某个老师不 OK
    • 不要 AI 味。如果你用 LLM 起草，自己重写一遍再提交。AI 给的"丝滑过渡句"和"完美总结段"恰恰是这个站最反对的语气
    • emoji 不用。除非内容性质特殊要求（如表情等）

    一点说明

    这个站不接受任何形式的赞助/广告/付费推广。如果你写的内容隐含商业意图（推某家公司的工具、某个收费课程），会被拒绝。

    如果你愿意把自己的名字留在站点（贡献者列表 / commit author），欢迎；如果你想匿名，也完全 OK——这个站从来就不是为了攒名字的。

    最后：贡献的门槛是低的，但贡献的标准是高的。报一个错别字也好，写一整个方向的指南也好，重要的是"准确、有用、有立场"，不是"多"。

    谢谢你愿意把你那段路画进这张地图里。
